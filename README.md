# Менеджер паролей

Самохостинговый менеджер паролей с шифрованием на стороне клиента. PHP + SQLite на сервере, vanilla jQuery в браузере, никаких сторонних сервисов, никаких CDN.

Все секреты шифруются в браузере мастер-ключом пользователя до отправки на сервер. Сервер видит и хранит только зашифрованные блобы - ни логинов, ни паролей, ни заметок он расшифровать не может.

## Демо

Статическая версия без сервера: [https://sovenov.github.io/simply_password_manager](https://sovenov.github.io/simply_password_manager)

Логин для входа: `sovenov` / `sovenov`. Все данные сохраняются только в `localStorage` вашего браузера и никуда не передаются.

## Возможности

- AES-256-GCM с ключом из PBKDF2-SHA256
- Группы с drag-and-drop сортировкой
- Избранное с сортировкой по времени добавления в избранное
- OTP/TOTP с генерацией кода и сканером QR-кода через камеру
- Импорт и экспорт в JSON
- Тёмная тема
- Глобальная видимость полей (логин/пароль/заметки/OTP) + точечный «глаз» у каждого поля
- Service Worker - работает offline после первой загрузки
- Шифрованный локальный кеш данных в `localStorage` - мгновенный старт без сети
- Поиск по записям
- Генератор паролей с настройкой длины и алфавита
- Админ-панель: модерация регистраций, сброс пароля юзеру, удаление учёток
- Авто-чистка сессий старше 30 дней при каждом входе
- CSRF-токены, Subresource Integrity на всех локальных ассетах, CSP без `unsafe-inline`

## Требования

**Сервер:**
- PHP 7.1+ с расширениями `pdo_sqlite` и `openssl`
- Желательно PHP-FPM (не обязательно)
- Apache или nginx
- HTTPS обязателен (без него браузер не отдаст Web Crypto API и Service Worker)
- (в целом, можете всё это закинуть даже на любой хостинг и оно будет работать даже там. Но лучше конечно свой vps/vds)

**Браузер пользователя:**
- Chrome 108+ / Firefox 114+ - основная цель заставить это всё работать на windows 7
- Safari iOS 11+ / Safari macOS 11+ / Chrome Android 49+ - поддерживаются
- Internet Explorer не поддерживается (нет Web Crypto API)

## Установка

1. Залить файлы на сервер.

2. Создать `configs/cfg.ini` по образцу [`configs/cfg.ini.example`](configs/cfg.ini.example) (если файла нет - см. ниже секцию «Конфигурация»).

3. Убедиться, что веб-сервер не отдаёт по прямой ссылке `configs/`, `DB/`, `bash_scripts/`, `up/`. Для Apache `.htaccess` уже лежат в этих папках. Для nginx добавить аналогичный `deny all` в конфиг.

4. Дать веб-серверу права на запись в `DB/` - туда будет создан `db.sqlite` при первом запросе.

5. Открыть сайт в браузере. БД и первый admin-аккаунт создадутся автоматически.

6. Войти с `default_admin_login` / `default_admin_password` из cfg.ini. В поле «Ключ шифрования логина и пароля» ввести `client_auth_key` из cfg.ini.

7. Сразу же поменять `default_admin_password` в админке.

## Конфигурация

`configs/cfg.ini`:

```ini
; Ключ для шифрования auth_key в БД (любая длинная строка)
db_encryption_key = "ваше-значение-1"

; Общий секрет, который клиент вводит при входе/регистрации
; в поле "Ключ шифрования логина и пароля"
client_auth_key = "ваше-значение-2"

; Клиентский секрет для шифрования мастер-ключа в localStorage
; (используется опцией "Сохранять ключ локально")
local_storage_secret = "ваше-значение-3"

; Учётные данные первого admin - создаются один раз при пустой БД
default_admin_login = "admin"
default_admin_password = "поменяйте-сразу-после-входа"
```

Все значения подхватываются на каждый запрос - после правки cfg.ini ничего перезапускать не нужно.

## Демо-версия

Папка [`DEMO/`](DEMO/) - статичная сборка, разворачивается на GitHub Pages, Netlify, Cloudflare Pages или просто открывается локальным `python -m http.server`.

- PHP не требуется.
- Логин жёстко зашит: `sovenov` / `sovenov`.
- Все данные хранятся только в `localStorage` пользователя - у каждого посетителя своя независимая «база».
- Шифрование работает по-настоящему: токены, мастер-ключ, PBKDF2 - всё то же, что в основной версии. Подделан только сетевой слой.

## Бэкап

`bash_scripts/backup.sh` запускается асинхронно после каждой успешной операции записи (см. `runBackupScriptAsync` в [api.php](api.php)). По умолчанию скрипт пустой - заполните своей стратегией: rsync, копия в S3, что угодно.

## Архитектура шифрования

- **Мастер-ключ** - вводится пользователем при каждом входе, в чистом виде хранится только в памяти вкладки (опционально шифруется и кладётся в localStorage через `local_storage_secret`).
- **Каждая запись** шифруется AES-256-GCM. Ключ выводится из мастер-ключа через PBKDF2-SHA256 со свежей солью, KDF-метаданные хранятся рядом с шифротекстом.
- **client_auth_key** - отдельный общий секрет для аутентификации. Логин и пароль пользователя шифруются им перед отправкой на сервер. Это даёт второй фактор: даже зная пароль, без `client_auth_key` войти нельзя.
- **CSRF-токен** - генерится при создании сессии, хранится в БД, требуется для всех POST-запросов кроме login/register.

## Что защищается, а что нет

**Защищает от:**
- Утечки БД с сервера - у атакующего только зашифрованные блобы без мастер-ключа.
- MitM на транспорте (под HTTPS) - данные доходят до сервера уже зашифрованными.
- XSS через сторонние скрипты - CSP без `unsafe-inline`, SRI на локальных ассетах.
- CSRF - токен сессии проверяется на каждом state-changing POST.
- Брутфорса слабых паролей пользователя на сервере - серверный `password_hash` через `PASSWORD_DEFAULT` PHP (bcrypt).

**НЕ защищает от:**
- Слабого мастер-ключа - 210 000 итераций PBKDF2 (ниже текущей рекомендации OWASP в 600 000), плюс PBKDF2 архитектурно слаб против GPU. Используйте мастер-ключ длиной 16+ случайных символов.
- Кейлоггера или вредоносного расширения в браузере пользователя.
- Скомпрометированного сервера - администратор может подменить отдаваемый JS и собрать мастер-ключ.
- Брутфорса входа на API - rate-limiting не реализован (делайте сами на nginx или apache).
